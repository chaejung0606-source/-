이 웹북의 사용 방법과 표기 기준

이 웹북은 혁신인재지원금·공간대여 플랫폼의 실제 소스 코드(버전 v1.1.189, 2026-07-09 기준)를 전수 확인하여 작성한 통합 운영 매뉴얼입니다. 행정 담당자의 플랫폼 점검·운영, 관리자·신청자 안내 자료로 사용할 수 있습니다.

구현 여부 표기 기준

이 문서의 모든 항목은 코드에서 확인된 사실만 기재하였으며, 다음 4단계로 구분해 표기합니다. 추측으로 "구현됨"이라고 적은 항목은 없습니다.

구현 · 코드 확인소스 코드에서 화면·로직·문구까지 확인된 기능. 단, 실제 클릭 테스트(런타임)는 운영 환경에서 별도 수행이 필요합니다(14장 체크리스트).

플랫폼 개요

1-1. 플랫폼 목적

이 플랫폼은 대학 사업단의 학생 지원금(혁신인재지원금) 신청·심사·지급 관리와 공간(강의실 등) 대여 신청·심사 관리를 온라인으로 처리하기 위해 만들어졌습니다. 종이 신청서·수기 대장을 대체하여 다음 업무를 처리합니다.

  • 지원금 신청 접수 — 8개 유형(아래 표)의 신청서를 온라인 폼으로 접수 (개인정보 동의·전자서명·증빙 파일 포함)
  • 2단계 신청 체계 — 활동 전 지원신청(사전신청) → 승인 → 활동 후 지원금 신청으로 이어지는 연계 구조 (유형에 따라 지원금 신청만 있는 경우도 있음)
  • 심사·검토 — 관리자의 검토 상태 관리(신청완료 → 검토중 → 보완요청/보완완료 → 심의필요 → 승인/반려)
  • 보완요청·재제출 — 관리자가 보완을 요청하면 신청자가 수정 후 재제출하고, 상태가 자동으로 '보완완료'로 표시
  • 지급 관리 — 최종 승인 금액 확정과 지급 상태(지급대기 → 지출결의중 → 지출완료 / 지급보류 / 환수대상) 관리
  • 서류 인계 — 프로그램 관리자 ↔ 지출관리자 간 서류 전달·반송(보완 요청) 흐름 관리
  • 공간대여 — 로그인 없이 신청하는 공간대여 접수, 예약 겹침 차단, 관리자 심사(승인/반려/보완), 구글 캘린더·시트 연동, 이용결과(명단·서명·사진) 제출
  • 운영 도구 — 신청폼 편집(폼 빌더), 신청 기간 설정, 자격증 목록 관리, 가상학과 명단 관리, 알림 발송, 엑셀 다운로드, 문서(PDF) 인쇄, 사이트 설정

1-3. 주요 업무 흐름

① 지원금 신청 흐름 (핵심 업무)

  1. 관리자(지출관리자)가 신청폼 편집에서 프로그램과 신청서 양식, 신청 기간을 등록합니다.
  2. 학생이 플랫폼에 접속해 회원가입·로그인합니다. (아이디 = 학번)
  3. (참여형 유형) 활동 전에 지원신청(사전신청)을 제출하고 관리자 승인을 받습니다.
  4. 학생이 지원금 신청서를 작성합니다 — 개인정보 수집·이용 동의, 유형·프로그램 선택, 폼 항목 입력, 증빙 파일 업로드, 전자서명.
  5. 제출 시 접수번호가 자동 발급되고 검토 상태는 신청완료가 됩니다.
  6. 프로그램 관리자가 담당 건을 검토한 뒤 지출관리자에게 보내기로 서류를 인계합니다.
  7. 관리자가 검토 상태를 변경합니다 — 검토중 / 보완요청(사유 메모) / 심의필요 / 승인 / 반려.
  8. 보완요청을 받은 학생은 수정 후 재제출하며, 상태가 자동으로 보완완료로 바뀌어 관리자가 재검토합니다.
  9. 승인 건은 지출관리자가 최종 승인 금액을 확정하고 지급 상태(지급대기 → 지출결의중 → 지출완료)를 관리합니다.
  10. 전 과정의 신청 데이터가 데이터베이스에 보관되고, 엑셀 다운로드·문서 인쇄(지출자료·심의요청서)로 행정 자료를 만듭니다.

② 공간대여 흐름

  1. 관리자가 대여가능공간(공간명·수용인원·사진)과 구글 캘린더 연동을 설정합니다.
  2. 이용자가 로그인 없이 공간대여 신청을 작성합니다 — 장소·일시·신청자 정보·동의. 이미 예약된 시간대는 자동 차단됩니다.
  3. 관리자가 승인 / 반려 / 보완요청으로 심사합니다. 승인 시 구글 캘린더·시트에 자동 반영됩니다(웹훅 설정 시).
  4. 이용 후 신청자가 연락처로 본인 신청을 조회해 이용결과(이용자 명단·서명·사진)를 제출합니다.

권한별 차이 설명

  • 관리자는 전체 신청내역을 볼 수 있는가? — 지출관리자는 전체를 조회합니다. 프로그램 관리자는 담당 프로그램으로 배정된 신청 건만 서버에서 걸러 내려받습니다(코드 확인: 신청 목록 API가 담당 프로그램 외 건을 제외). 구현 · 코드 확인
  • 신청자는 본인 신청만 보는가? — 마이페이지는 로그인한 본인 계정의 신청만 조회합니다. 데이터베이스 행 수준 보안(RLS)으로 타인 데이터 접근이 차단됩니다. 구현 · 코드 확인
  • 신청 후 수정·취소가 가능한가? — 제출 후 임의 수정 화면은 없으며, 보완요청 상태일 때만 마이페이지의 [수정 후 재제출] 버튼으로 수정·재제출할 수 있습니다. 신청 취소는 상태와 무관하게 마이페이지에서 가능하며(확인창 2회), 취소 내역으로 이동합니다. 지급 완료 건 취소 시 별도 처리 필요 — 운영상 주의.
  • 관리자는 신청 상태를 변경할 수 있는가? — 가능합니다. 검토 상태·안내 메모는 모든 관리자, 승인 금액·지급 상태는 지출관리자만 변경할 수 있고, 프로그램 관리자가 시도하면 서버가 403(권한 없음)으로 거부합니다. 구현 · 코드 확인
  • 권한 없는 사용자가 관리자 화면에 접근하면? — 관리자 API는 서명된 관리자 세션 쿠키를 검증해 401을 반환하고, 화면은 데이터가 비어 관리 기능을 사용할 수 없습니다. 프로그램 관리자가 부여받지 않은 메뉴의 API를 호출해도 401/403으로 차단됩니다. 또한 신청 목록·신청자 정보 화면은 진입할 때마다 관리자 비밀번호를 재확인합니다(개인정보 보호 게이트). 화면 단의 메뉴 숨김 + 서버 단의 API 검증 이중 구조입니다.

9장

데이터 흐름 및 저장 구조

데이터는 Supabase(PostgreSQL 데이터베이스 + 파일 저장소)에 보관됩니다. 신청자는 행 수준 보안(RLS)으로 본인 데이터만 접근할 수 있고, 관리자 작업은 서버를 통해서만 처리됩니다.

9-2. 데이터 생성·수정·삭제 시점 정리

  • 신규 생성 — 회원가입(계정·프로필), 신청서 임시저장/제출(신청 행 — 접수번호는 제출 시 연도-순번으로 DB가 자동 부여, 삭제된 번호는 재사용하지 않음), 공간대여 접수, 알림 발송.
  • 수정 — 임시저장 갱신, 보완 재제출(같은 행 갱신 + 상태 자동 변경), 관리자 상태·메모·금액 저장, 프로필 수정(이후 신청서 자동 입력에 반영), 공간대여 일정 수정.
  • 삭제인가, 상태 변경인가 — 신청 취소는 실제 삭제가 아니라 취소 플래그 기록입니다(취소 일시·IP 보관, 취소 목록에서 계속 조회 가능). 실제 삭제는 ① 테스트 신청 삭제(지출관리자) ② 회원 탈퇴 시 미지급 건 ③ 공간대여 신청 삭제뿐입니다. 지출완료 건은 탈퇴해도 정산 증빙을 위해 보존됩니다.
  • 연결 구조 — 신청은 신청자 계정 ID와 연결(탈퇴 시에도 지출완료 기록은 신청서 내용 유지). 신청 유형·프로그램명으로 프로그램과 연결되고, 이 값으로 프로그램 관리자의 담당 범위가 판정됩니다.
  • 상태 변경 이력 — 미구현. 현재 상태값만 저장되며 변경 이력(누가·언제)은 남지 않습니다.
